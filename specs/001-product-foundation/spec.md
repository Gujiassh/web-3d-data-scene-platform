# PRD：Web 3D 数据场景平台 MVP

> 状态：Product Design Complete / Validation Pending
> 版本：1.0
> 日期：2026-07-13
> 单 Studio 修订：2026-07-15；当前实施与路线由 `../005-single-studio-data-binding/spec.md` 接管

## 1. 摘要

MVP 提供一条完整的 Web 3D 业务场景生产链路：导入已有模型，在浏览器中组织场景，
为场景实体绑定实时数据和显示规则，然后将同一份场景文档发布到可嵌入 Viewer。

固定 M0 工厂 scene/GLB 是首个合同和运行时 fixture，用于验证设备状态、告警定位和恢复；
它不构成第二个产品应用，所有核心数据结构和运行时能力必须保持行业无关。

## 2. 目标

- 证明 Editor、场景文档、数据绑定和 Viewer 能形成闭环。
- 展示复杂 React 前端与 Three.js 场景之间清晰的状态边界。
- 提供可审阅、可测试、可部署的开源求职项目。
- 让目标开发者无需编写场景内部代码即可完成基础状态映射。
- 为后续仓库、园区、机房和商品配置示例保留扩展空间。

## 3. 非目标

- 不提供在线建模、CAD、拓扑编辑或模型生成。
- 不实现真实 PLC、OPC UA、MES 或设备控制。
- 不实现多人实时协作、云端组织管理或权限系统。
- 不提供游戏逻辑、关卡系统或完整物理引擎集成。
- 不承诺导入所有 DCC 软件和私有模型格式。
- 不在 MVP 中实现可视化插件市场。

## 4. 用户角色

### P1：前端工程师

希望把 3D 场景嵌入业务系统，并通过稳定 API 接收选择事件、更新数据源和控制镜头。

### P2：解决方案工程师

希望通过配置而不是修改 Three.js 代码，完成模型摆放、实体映射、告警规则和客户演示。

### P3：实施或演示维护者

希望打开已有项目，调整模型布局、标注和绑定，并安全地发布新版本。

## 5. 关键用户流程

### Flow A：创建数据驱动场景

1. 用户创建本地项目并导入有效的 GLB/GLTF。
2. 系统展示资产、场景树和 3D 视口。
3. 用户选择对象并调整位置、旋转、缩放和业务实体标识。
4. 用户绑定数据路径并配置状态显示规则。
5. 用户启动模拟数据，预览场景变化和连接状态。
6. 用户导出版本化场景文档。

### Flow B：嵌入 Viewer

1. 开发者在示例宿主应用中安装或加载 Viewer。
2. Viewer 读取导出的场景文档和模型资产。
3. 宿主应用提供模拟或 WebSocket 数据适配器。
4. Viewer 执行相同规则，并将实体选择事件返回宿主应用。
5. 数据连接中断后，Viewer 显示陈旧或离线状态，恢复后自动同步最新快照。

### Flow C：在 Studio Run 定位故障对象

1. 固定参考场景中的对象收到故障状态。
2. 对象按规则显示故障视觉状态，并产生告警事件。
3. 用户从 Studio Run 的告警列表选择该事件。
4. 同一 Viewer 定位并选中对应 Target，Run UI 展示当前值和连接状态。
5. 数据恢复后，场景和告警状态按最新输入更新，SceneDocument 保持不变。

## 7. 概念数据模型

此处只定义产品语义，不构成最终 TypeScript 或持久化合同。

| 概念          | 责任                                                 |
| ------------- | ---------------------------------------------------- |
| SceneDocument | 保存 schemaVersion、资源、实体、绑定、规则和视图配置 |
| Asset         | 引用模型文件及其元数据，不承载业务实时状态           |
| Entity        | 保存稳定 ID、层级、变换、可编辑属性和 assetRef       |
| DataSource    | 描述适配器类型、连接配置和健康状态语义               |
| Binding       | 将 entityId/property 关联到外部数据路径              |
| Rule          | 将输入值转换为颜色、可见性、标签或告警结果           |
| Annotation    | 将 3D 锚点关联到宿主应用展示内容                     |

运行时数据、当前选择、相机过渡、连接实例和瞬时告警不得写回 SceneDocument。

## 8. 验收场景

### AC-001：项目往返

给定一个包含层级、变换、绑定和规则的项目，导出后重新导入，所有稳定实体 ID、
父子关系、变换值、绑定路径和规则结果保持一致。

### AC-002：安全导入失败

给定无效或不支持的模型文件，导入失败时展示可操作错误信息，当前场景文档和已加载
资源保持不变。

### AC-003：撤销重做

给定移动、重命名和删除三个连续操作，执行三次撤销后恢复初始状态，执行三次重做后
恢复最终状态。

### AC-004：实时状态映射

给定实体绑定到 `machine.status`，当适配器依次发送 `running`、`fault`、`offline` 时，
实体在参考环境中 100ms 内应用对应规则，且不改变持久化场景文档。

### AC-005：断线和恢复

给定 Viewer 已连接 WebSocket，连接断开超过可配置陈旧阈值后显示离线状态；重新连接
并收到快照后恢复最新状态，不回放会造成错误最终状态的过期事件。

### AC-006：Studio Run/Viewer 一致性

给定同一场景文档、数据快照和规则集合，Studio Run 与发布 Viewer 对每个 Target
产生相同的规则结果。

### AC-007：宿主集成

给定 React 示例应用，开发者可以加载场景文档、提供数据适配器、监听实体选择事件，
并由宿主 UI 控制 Viewer 聚焦指定实体。

### AC-008：固定参考场景

给定固定 M0 scene/GLB 和 Mock 数据，Studio Run 能演示正常运行、故障告警、离线和恢复，
告警选择可以定位正确 Target，fixture 内部工厂语义不进入平台固定 UI。

### AC-009：场景树与视口同步

给定包含父子层级的实体，从场景树或视口选择实体时，两处显示同一个稳定实体 ID；
隐藏实体不再渲染但仍保留在文档中，锁定实体不能通过视口变换控件修改。

### AC-010：版本不兼容

给定高于当前支持范围的 schemaVersion，Editor 和 Viewer 拒绝加载，明确显示收到的
版本和支持范围，并且不修改当前已加载项目。

### AC-011：诊断状态

给定模型资源失败、数据连接断开或规则求值错误，诊断界面分别显示错误类型、相关资源
或实体以及可执行的排查信息，同时不展示数据源凭据。

## 10. 非功能需求

| ID      | 要求                                                                            |
| ------- | ------------------------------------------------------------------------------- |
| NFR-001 | Studio 支持当前主流桌面 Chromium、Firefox 和 Safari；Viewer 支持桌面和平板布局  |
| NFR-002 | 在约定参考设备和基准场景中，Viewer 稳态帧率目标为 60 FPS，1% low 不低于 30 FPS  |
| NFR-003 | 在包含 300 个实体和 100 个活动绑定的基准场景中，选择反馈 p95 低于 100ms         |
| NFR-004 | 数据更新和渲染解耦；高频事件允许合并，但最终状态不得丢失                        |
| NFR-005 | 数据源凭据不得写入导出的场景文档，日志不得输出敏感连接参数                      |
| NFR-006 | 场景文档必须版本化，并为不兼容版本提供显式拒绝或迁移路径                        |
| NFR-007 | 2D 编辑器 UI 支持键盘操作、可见焦点和文本标签；场景树提供 Canvas 操作的替代入口 |
| NFR-008 | 无效模型、资源加载失败、数据断线和规则错误必须具有独立可诊断状态                |
| NFR-009 | 相同场景文档和数据快照必须产生确定性的规则结果                                  |

性能数字是待验证的工程目标。基准设备、浏览器、模型大小、三角面数和测量脚本必须在
实施计划中固定后，才能判断通过或失败。

## 11. MVP 垂直切片

当前单 Studio 垂直切片只证明以下闭环：

1. 在 Studio 导入固定小型 GLB，并使用导入资产根 Target。
2. 分配稳定 business ID。
3. 创建确定性 Mock source 并选择规范 JSON Pointer。
4. 配置 `running`、`fault`、`offline` 三种 color/alarm 规则。
5. 在同一 Studio 中进入 Run，验证连接、当前值、选择、告警、离线和恢复。
6. 导出并重导 SceneDocument/archive，验证 ID、pointer、规则和 hash 不变。

布局、热点交互、发布嵌入和性能/可用性/开源门禁分别由 006、007、008、009 接管。

## 12. 风险与依赖

| 风险                             | 严重度 | 应对                                       |
| -------------------------------- | ------ | ------------------------------------------ |
| 范围膨胀为通用 DCC/游戏编辑器    | 高     | 以数据绑定闭环作为唯一 MVP 主线            |
| 场景文档与 Three.js 对象结构耦合 | 高     | 保持领域模型独立，使用适配层投影到运行时   |
| Studio Run 和 Viewer 行为漂移    | 高     | 共享规则执行和场景解释核心，并做一致性测试 |
| AI 生成模型风格和拓扑不稳定      | 中     | MVP 使用少量模块化自有或 CC0 资产          |
| 大模型和实时数据造成性能问题     | 中     | 建立固定基准、预算和诊断工具               |
| 直接连接外部数据引入安全问题     | 中     | MVP 仅使用模拟源，凭据由宿主应用提供       |
| 开源项目缺少真实使用反馈         | 高     | 在扩展功能前进行外部开发者任务测试         |

## 13. 已解决的产品问题

1. 主要投递目标为通用高级前端岗位，3D 可视化是差异化能力。
2. Viewer 第一版提供框架无关 API，并提供 React 包装层。
3. 固定 M0 factory fixture 保留可复现生成器、CC0 许可和 hash oracle，不作为产品应用。
4. Studio 是唯一产品前端；独立 Factory Demo 由 feature 005 supersede。
5. REST 轮询保留在 Could，feature 005 先完成 Mock，WebSocket 后续扩展。
6. 发布和最小嵌入宿主由 feature 008 验收。

决策依据与边界见 `docs/ssot/product-decisions.md`。
