# 需求质量检查表

> 目标：检查需求本身是否完整、明确、一致、可测，而不是检查实现。

## 完整性

- [x] 已定义主要用户及其核心工作任务。
- [x] 已定义目标、非目标和 MVP 边界。
- [x] 已覆盖单 Studio、场景文档、数据绑定、Viewer 库和参考 fixture。
- [x] 已覆盖空状态、导入失败、数据断线和版本不兼容。
- [x] 已定义功能需求与非功能需求。
- [x] 已定义可独立验证的核心验收场景。
- [x] 每项 Must 功能需求均映射到至少一个验收场景。
- [ ] 已用外部目标用户证据验证问题陈述。
- [x] 已固定性能参考设备、浏览器、模型和数据负载。
- [x] 已确认模型资产生成方案与许可证策略。

## 明确性

- [x] 在线建模、真实工业接入、多人协作和物理仿真明确排除在 MVP 外。
- [x] 运行时瞬时状态与持久化 SceneDocument 明确分离。
- [x] 工厂数据被定义为测试 fixture，而不是产品表面或平台领域模型。
- [x] Editor 与 Viewer 必须共享规则语义。
- [x] “发布”的文件产物和 API 形式已在技术设计中固定。
- [x] “框架无关 Viewer”的支持范围已在技术设计中固定。

## 一致性

- [x] 产品定义、定位记录和 PRD 使用相同的一句话定位。
- [x] Must 功能覆盖三条关键用户流程。
- [x] 非功能要求没有要求 MVP 接入真实工业设备。
- [x] 数据绑定差异点在定位、范围和验收场景中一致。
- [x] 技术设计和 Schema 均不持久化 Three.js 实例或运行时状态。
- [x] M0 已用 workspace 依赖边界、Schema negative case 和确定性序列化测试验证代码没有
      突破持久化/运行时边界。

## 可测性

- [x] 项目导入导出定义了往返一致性验收。
- [x] 撤销重做定义了有顺序的验证场景。
- [x] 数据规则定义了输入、输出和响应时间目标。
- [x] 数据断线定义了陈旧、离线和恢复语义。
- [x] Editor/Viewer 一致性定义了相同输入的对比方式。
- [x] 场景树同步、版本拒绝和诊断状态具有独立验收场景。
- [x] 性能要求已定义正式基准负载、参考环境和测量方式。
- [x] 外部开发者 15 分钟集成目标已定义可用性测试脚本。

## 进入发布验收前的门禁

- [ ] 完成 3 至 5 名目标开发者的需求或任务访谈。
- [x] 已评审并确认 MVP Must/Should/Could 列表。
- [x] 已确认第一版演示资产方案和许可证。
- [x] 已形成 SceneDocument、Data Adapter 和 Viewer API 技术设计。
- [x] M0 垂直合同切片已通过本地工程与 Chromium 浏览器验收。
- [ ] 在 005 和 008 主流程可用后，使用当前真实原型执行外部目标开发者任务测试。
