# 产品定义：Web 3D 数据场景平台

> 状态：Product Design Complete / Validation Pending
> 日期：2026-07-13
> 证据等级：产品假设，尚未经过外部用户验证

## 一句话定位

一个开源、可自托管、可嵌入的 Web 3D 场景搭建平台，让前端团队导入已有
GLB/GLTF 模型、绑定实时业务数据，并发布可嵌入现有系统的 3D Viewer。

## 需要解决的问题

需要交付 3D 可视化功能的前端团队，通常必须为每个项目重复实现模型加载、
场景组织、对象拾取、业务实体映射、状态着色、告警定位、数据接入和页面嵌入。
通用 3D 编辑器主要解决视觉创作，工业数字孪生平台又往往过重、云服务绑定或
成本较高，两者之间缺少一个面向前端开发者的轻量开源层。

以上问题陈述目前是产品假设，不代表已有市场需求证据。

## 目标用户

### 主要用户

需要在业务系统中交付 3D 可视化能力的前端工程师、解决方案工程师和小型产品团队。

### 次要用户

- 制作工业、仓储、园区或展厅演示的独立开发者
- 需要维护 3D 场景但不负责建模的实施人员
- 希望快速验证 Three.js 业务方案的技术团队

## 核心工作任务

当团队已经拥有 3D 模型，并需要把模型变成一个可运行的业务页面时，他们希望：

1. 不编写场景胶水代码也能完成模型摆放和对象组织。
2. 为模型对象分配稳定的业务实体标识。
3. 将实体绑定到 REST、WebSocket 或模拟数据。
4. 用规则配置颜色、可见性、文本和告警状态。
5. 将同一份场景文档发布为可嵌入的运行时 Viewer。

## 核心价值

- **减少重复开发**：沉淀通用编辑、绑定和运行时能力。
- **保持业务可控**：支持自托管和版本化场景文档。
- **降低集成成本**：运行时可嵌入现有前端应用，而不是替换现有系统。
- **分离职责**：模型资产、场景配置、实时数据和宿主 UI 保持独立。

## 产品边界

### 平台负责

- 导入和管理已有 Web 3D 资产
- 组织、编辑和序列化场景
- 建立场景实体与业务数据之间的映射
- 执行确定性的可视化规则
- 提供预览、发布和嵌入式 Viewer
- 提供可观察的数据连接状态和渲染性能信息

### 平台不负责

- 在线建模、雕刻、CAD 或 DCC 工具替代
- 真实 PLC、OPC UA、MES 或 IoT 网关实现
- 游戏引擎、物理沙盒或关卡系统
- 第一版中的多人实时协作
- 第一版中的云端账号、计费和企业权限系统
- 为所有行业提供完整模型资产库

## 产品组成

1. **Editor**：资产、场景树、视口、属性、数据绑定和规则编辑。
2. **Scene Document**：版本化、可迁移、可导入导出的领域中立 JSON 文档。
3. **Data Adapter**：统一模拟数据、静态 JSON、REST 轮询和 WebSocket 的输入语义。
4. **Viewer Runtime**：只读加载场景、消费数据并执行可视化规则。
5. **Integration Layer**：框架无关运行时接口，并提供 React 包装层和嵌入示例。
6. **Reference Template**：一条小型工厂流水线，用于证明状态同步、告警和物料流转。

## MVP 成功门槛

这些是交付门槛，不是已验证的市场指标：

- 新开发者能依照文档在 15 分钟内运行示例并嵌入 Viewer。
- 同一份场景文档在 Editor 预览和独立 Viewer 中具有一致语义。
- 工厂示例能稳定展示运行、待机、故障、离线和恢复状态。
- 场景导出再导入后，实体标识、层级、变换、绑定和规则保持一致。
- 参考性能场景满足 PRD 中定义的帧率和交互延迟目标。
- 仓库包含架构说明、数据协议、自动化测试和可访问的在线演示。

## 当前关键假设

| 假设                                                          | 风险 | 验证方式                                       |
| ------------------------------------------------------------- | ---- | ---------------------------------------------- |
| 前端团队愿意采用独立的场景文档和 Viewer，而不是继续项目内定制 | 高   | 访谈 5 名目标开发者，并完成 3 次可用性测试     |
| 数据绑定是相对通用编辑器更有价值的差异点                      | 高   | 用薄切原型与 Three.js Editor/Spline 类流程对比 |
| 自托管和开源对目标用户有明确价值                              | 中   | 收集访谈中的安全、部署和采购约束               |
| 工厂示例足以表达能力且不会让平台看起来仅限工业                | 中   | 再增加一个非工业的轻量示例验证复用性           |
| 使用已有模型而非内置建模能被目标用户接受                      | 中   | 观察测试用户是否在导入和模型准备阶段受阻       |

## MVP 已采纳决策

- 求职目标优先于商业化目标。
- 平台能力保持行业无关，工厂只是首个参考模板。
- 第一版采用 MIT 许可证，模型资产单独记录许可证。
- 第一版桌面编辑优先，Viewer 必须支持平板和常见桌面尺寸。
- 第一版不依赖真实后端，以本地文档和模拟 WebSocket 完成垂直验证。

完整决策与理由见 [产品决策记录](product-decisions.md)。
