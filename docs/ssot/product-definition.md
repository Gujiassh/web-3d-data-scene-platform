# 产品定义：SceneWeave

> 状态：Product Design Complete / Validation Pending
> 初始日期：2026-07-13
> 当前产品拓扑：2026-07-15 由 `specs/005-single-studio-data-binding` 修订
> 证据等级：产品假设，尚未经过外部用户验证

## 一句话定位

SceneWeave 是一个开源、可自托管、可嵌入的 Web 3D 场景搭建平台，让前端团队在一个 Studio 中导入
已有 GLB/GLTF 模型、绑定实时业务数据、预览规则结果，并发布可嵌入现有系统的 Viewer。

## 需要解决的问题

需要交付 3D 可视化功能的前端团队，通常必须为每个项目重复实现模型加载、场景组织、
对象拾取、业务实体映射、状态着色、告警定位、数据接入和页面嵌入。通用 3D 编辑器主要
解决视觉创作，工业数字孪生平台又往往过重、云服务绑定或成本较高，两者之间缺少一个
面向前端开发者的轻量开源层。

以上问题陈述目前是产品假设，不代表已有市场需求证据。

## 目标用户

### 主要用户

需要在业务系统中交付 3D 可视化能力的前端工程师、解决方案工程师和小型产品团队。

### 次要用户

- 制作工业、仓储、园区或展厅演示的独立开发者
- 需要维护 3D 场景但不负责建模的实施人员
- 希望快速验证 Three.js 业务方案的技术团队

## 核心工作任务

当团队已经拥有 3D 模型，并需要把模型变成一个可运行的业务页面时，他们希望：

1. 在一个 Studio 中完成模型导入、场景组织和运行预览。
2. 为模型对象分配稳定的业务实体标识。
3. 将实体绑定到 Mock、WebSocket 或后续支持的数据源。
4. 用声明式规则配置颜色、可见性、文本和告警状态。
5. 将同一份场景文档发布为可嵌入的运行时 Viewer。

## 核心价值

- **减少重复开发**：沉淀通用编辑、绑定和运行时能力。
- **保持业务可控**：支持自托管和版本化场景文档。
- **降低集成成本**：运行时可嵌入现有前端应用，而不是替换现有系统。
- **分离职责**：模型资产、场景配置、实时数据、运行时状态和宿主 UI 保持独立。

## 产品边界

### 平台负责

- 导入和管理已有 Web 3D 资产
- 组织、编辑和序列化场景
- 建立场景 Target 与业务数据之间的映射
- 执行确定性的可视化规则
- 在 Studio Run 中预览连接、当前值、颜色、告警和诊断
- 发布版本化场景并提供可嵌入 Viewer
- 提供可观察的数据连接状态和渲染性能信息

### 平台不负责

- 在线建模、雕刻、CAD 或 DCC 工具替代
- 真实 PLC、OPC UA、MES 或 IoT 网关实现
- 游戏引擎、物理沙盒或关卡系统
- 第一版中的多人实时协作
- 第一版中的云端账号、计费和企业权限系统
- 为所有行业提供完整模型资产库
- 固定的工厂运营面板、设备控制或告警确认业务

## 产品组成

1. **Studio**：唯一用户可见前端，负责本地项目、场景编辑、数据绑定、规则和 Run 预览。
2. **Scene Document**：版本化、可迁移、可导入导出的领域中立 JSON 合同。
3. **Viewer Runtime**：框架无关地加载场景、消费数据并执行可视化规则。
4. **React Wrapper**：将 Runtime 生命周期和命令句柄映射到 React 宿主。
5. **Reference Fixtures**：确定性的 GLB、SceneDocument、manifest 和 Mock 数据，用于合同、
   archive、runtime 和浏览器回归，不构成第二个产品应用。

Studio 是唯一产品前端。Runtime 和 React Wrapper 继续作为可复用库边界，并在发布阶段通过
最小嵌入示例证明集成能力。独立 Factory Demo 已由 feature 005 supersede；其通用运行时证据
迁入 Studio Run 和自动化测试后，原应用已从仓库删除。

## 成功门槛

这些是交付门槛，不是已验证的市场指标：

- 新开发者能依照文档启动 Studio，并在一个应用中完成 Edit 和 Run 主流程。
- 同一份场景文档在 Studio Run 和发布 Viewer 中具有一致语义。
- 固定参考 fixture 能稳定证明运行、故障、离线、恢复、选择和告警状态。
- 场景导出再导入后，实体标识、层级、变换、绑定和规则保持一致。
- 参考性能场景满足 PRD 中定义的帧率和交互延迟目标。
- 仓库包含架构说明、数据协议、自动化测试、嵌入教程和可访问的在线 Studio。

## 当前关键假设

| 假设                                                          | 风险 | 验证方式                                       |
| ------------------------------------------------------------- | ---- | ---------------------------------------------- |
| 前端团队愿意采用独立的场景文档和 Viewer，而不是继续项目内定制 | 高   | 访谈 5 名目标开发者，并完成 3 次可用性测试     |
| 数据绑定是相对通用编辑器更有价值的差异点                      | 高   | 用薄切原型与 Three.js Editor/Spline 类流程对比 |
| 自托管和开源对目标用户有明确价值                              | 中   | 收集访谈中的安全、部署和采购约束               |
| 单 Studio Edit/Run 比双应用演示更容易理解                     | 中   | 记录首次任务的入口错误和完成率                 |
| 使用已有模型而非内置建模能被目标用户接受                      | 中   | 观察测试用户是否在导入和模型准备阶段受阻       |

## 当前路线

1. `005-single-studio-data-binding`：单 Studio、业务映射、规则编辑和 Mock Run 闭环。
2. `006-scene-layout`：场景布局、层级组织、对齐和吸附。
3. `007-hotspots-interactions`：精确表面热点、标注和持久化交互。
4. `008-publish-embed`：发布产物、最小宿主示例和嵌入文档。
5. `009-performance-usability-open-source`：固定基准、外部可用性和开源发布门禁。
