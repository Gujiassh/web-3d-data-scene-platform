# 产品决策记录

> 状态：Accepted for MVP
> 日期：2026-07-13
> 适用范围：产品设计与 MVP 实现

## PD-001：主要目标用户

**决策：** 主要用户是需要交付 Web 3D 业务功能的前端工程师和解决方案工程师，
不是工厂操作员或专业 3D 设计师。

**理由：** 求职目标要求项目能体现通用前端工程能力；操作员体验通过参考应用验证，
但不应主导平台内核。

## PD-002：产品定位

**决策：** 产品是数据驱动的 Web 3D 场景搭建与嵌入平台，不是在线建模器、游戏引擎
或工业 IoT 后端。

**理由：** 通用场景编辑器已有成熟竞品，数据绑定和可嵌入运行时才是本项目的有效切口。

## PD-003：产品表面

**决策：** MVP 包含四个表面：Studio、Viewer Runtime、React Wrapper 和 Factory Demo。

- Studio 负责本地项目、场景编辑、绑定、规则和预览。
- Viewer Runtime 负责加载、解释、渲染和宿主事件。
- React Wrapper 提供声明式集成与命令句柄。
- Factory Demo 证明实时状态、告警定位和宿主 UI 集成。

## PD-004：运行时框架边界

**决策：** Viewer Runtime 基于 TypeScript 和 Three.js，公开框架无关的容器式 API；
React 只作为第一方包装层，不进入场景文档协议。

**理由：** 这既证明 React 集成能力，也避免场景资产和业务数据被锁定到 React 组件树。

## PD-005：本地优先

**决策：** Studio MVP 使用浏览器本地存储和文件导入导出，不实现账号和云项目服务。

**理由：** 可以静态部署、降低后端范围，并让开源评审者直接运行完整闭环。

## PD-006：发布产物

**决策：** Studio 输出标准 ZIP 归档，包含 `manifest.json`、`scene.json` 和 `assets/`。
Viewer 同时支持加载解压后的 `scene.json` URL 和归档 Blob/File。

**理由：** 使用公开格式而非私有二进制；既方便版本控制和检查，也支持静态托管。

## PD-007：实时数据与凭据

**决策：** SceneDocument 仅记录数据源逻辑 ID、适配器类型和非敏感选项；URL、令牌和
认证材料由宿主应用在运行时注入。

**理由：** 场景文档需要可分享，不能承担秘密存储或真实工业网络连接职责。

## PD-008：数据路径和规则

**决策：** 数据路径采用 JSON Pointer；规则仅使用声明式运算符和效果，不允许执行
任意 JavaScript。

**理由：** 避免点号路径歧义、保证确定性，并缩小嵌入场景的安全面。

## PD-009：资产策略

**决策：** Factory Demo 使用 Blender Python 生成的模块化低模资产，提交生成脚本，
导出 GLB 使用 CC0-1.0；平台代码使用 MIT。

**理由：** 资产来源和再分发权清晰，AI 能稳定维护程序化模型，也不会依赖来源不明的
高精度机械模型。

## PD-010：响应式范围

**决策：** Studio 支持宽度不低于 1280px 的桌面环境；Viewer 支持 768px 以上平板
和桌面。更窄设备提供只读兼容提示，不提供完整编辑能力。

**理由：** 复杂场景编辑需要稳定的多面板尺寸，强行压缩到手机会损害核心工作流。

## PD-011：首版数据适配器

**决策：** MVP 只提供 Mock 和 WebSocket；REST 轮询保留为 Could。

**理由：** 两种适配器已经可以验证静态演示、实时更新、断线、重连和快照恢复。

## PD-012：求职交付标准

**决策：** 完整交付必须同时包含在线演示、开源仓库、架构文档、协议文档、自动化测试、
性能基准和一条可复现的嵌入教程。

**理由：** 单张效果图或单一 3D 场景无法证明产品工程能力。
